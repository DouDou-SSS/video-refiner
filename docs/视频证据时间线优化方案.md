# 视频证据时间线优化方案

## 背景与目标

视频炼化当前擅长博主主页批量解析、下载阶梯、文案可靠性校验、任务恢复，以及向 VideoAutomation 导出结构化起号基底。

本次优化借鉴 XLB Storyboard 的工作方法，而不复制其代码、Prompt 或产品界面。核心目标是：先形成可追溯的视频视觉证据时间线，再让 5 个单视频维度和第 6 维 Benchmark Intelligence 基于同一份证据分析与汇总。

## 不变的原则

- 软字幕、底部 OCR、Whisper 仍是文案事实来源；没有可靠文案不得进入炼化。
- 静态画面不能被当作声音、完整台词或运镜的事实来源。
- 云端模型拒答、结构化输出损坏、证据覆盖不足时，任务必须等待自动重试或清晰失败，不能以占位内容完成或导出。
- 保留现有 5 个单视频维度、第 6 维 creator-level Benchmark Intelligence、旧版 Markdown 和现有目录兼容性。
- 导出给 VideoAutomation 时不带原始视频、完整转写、缓存、Cookie、Token 或 API Key。

## 七项实现

### 1. 一次性视频证据时间线

每个成功视频在 `evidence/{video_id}.visual_timeline.json` 生成独立证据文件。文件记录视频时长、场景变化曲线摘要、时间线节点范围、关键帧、图像观察、底部 OCR、关联文案片段、稳定 evidence ID 与置信度。节点分为 `evidence_window`（抽样分析窗口）与 `detected_cut_segment`（由检测切点围出的切段）；前者不等于真实镜头时长。

该文件只记录相对文件名和结构化文字，不复制视频；帧图仍留在原始数据目录，便于本机复核和清理。

### 2. 覆盖帧与事件帧联合选样

保留现有默认抽帧策略：10 分钟内每秒一帧，10 分钟外每 5 秒一帧；用户自定义间隔仍优先。

新增 FFmpeg 场景变化检测，在连续高分区间选择峰值作为切分候选。同时保留均匀覆盖、开头、结尾、字幕明显变化等候选。默认镜头预算：短视频 8-12 段、1-5 分钟 16-24 段、5-10 分钟 24-36 段；超过预算时按时间分桶保留高分事件和覆盖帧。

### 3. 图文时间对齐

文案提取额外写入时间线 sidecar。Whisper 使用原始 segment 的 `start/end`；分段转写保留绝对时间；软字幕与 OCR 无法提供可靠时间时必须显式标为低置信度，不伪造精确时间。

每个镜头只关联其时间窗口附近的文案片段和 OCR 文本，避免把整条视频的文字塞给单个视觉判断。

### 4. 5 个单视频维度复用证据

单视频蒸馏不再只均匀抽取最多 20 张帧图。它读取证据时间线，按节点选取有代表性的关键帧，并在 Prompt 中提供稳定 evidence ID、时间范围和对应文案。要求关键视觉、剪辑与脚本结论引用 evidence ID。精确镜头时长、转场频率等结论只可使用具备精确时序资格的 `detected_cut_segment`。

运镜判断仅在存在相邻帧窗口时作出，并以“观察到”或“不确定”表达。音频和台词只引用转写/OCR/字幕证据。

### 5. Benchmark 与导出增强

Video Card 写入 `visual_timeline_ref`、`evidence_coverage` 和稳定 evidence ID。Benchmark 的 Card/Notes 与 creator-level 文档只能使用通过证据门禁的视频。

导出包复制轻量时间线 JSON，不复制图片、视频或完整文案。导出核验检查时间线与 Card 的引用一致、相对路径安全、核心字段完整且无模型拒答。

### 6. 只读证据时间线界面

任务详情新增“证据时间线”区域：显示视频时长、视觉覆盖率、镜头卡片、关键帧、关联文案、OCR、证据 ID 与置信度。

第一版只做查看和文件打开，不做完整手工分镜编辑器。后续可以在同一数据结构上增加“重跑单镜头”。

### 7. 模型能力层与长视频策略

现有图片视觉能力继续作为必须条件。原生视频输入必须由供应商能力测试确认 `supports_video_input` 后才允许启用；仅“支持图片”不等于“支持上传视频”。

第一版不让原生视频输入替代本地证据层。长视频按供应商限制和固定片段策略处理，保留绝对时间与重叠区间，不通过比例缩放强行修正模型时间戳。

## 实施顺序

1. 新增证据时间线模块和测试，不改变旧输出。
2. 在文案提取中写入时间 sidecar，构建图文对齐。
3. 接入单视频 5 维 Prompt 与证据引用，保留原 Markdown 文件名。
4. 将证据覆盖率写入 Benchmark Card、Notes、检索索引和导出包。
5. 增加任务详情的只读证据时间线 API 与界面。
6. 同步 README、SKILL、Web 使用文档和 manifest。

## 质量门禁与验收标准

- 每个完成视频必须有可靠文案、至少一个关键帧、可解析的证据时间线与稳定 evidence ID。
- 每个视觉/剪辑结论必须能回溯到具体视频时间范围；没有证据时必须表达不确定，不得补写事实。
- 模型拒答、坏 JSON、空 Card、无证据引用、时间线缺失均阻止 Benchmark 完成和 VideoAutomation 导出。
- 5 个单视频维度复用同一份证据时间线，而不是重复独立理解同一批帧。
- `retrieval_index.json`、Card、Notes 和导出包不含本机绝对路径、原始视频、完整转写和密钥。
- 后端测试覆盖：场景峰值选取、时间对齐、覆盖率门禁、Card/导出引用一致性。
- 用代表性视频集验证：高文字画面、静态讲解、高切镜、无硬字幕、长视频、模型拒答。只有质量门禁通过的任务才标记完成。

## VideoAutomation 复核补充（2026-06-20）

- 每个节点新增 `segment_type`、`boundary_source` 与 `boundary_confidence`，避免把抽样窗口误读为真实切镜段。
- `quality` 同时记录 `observation_coverage`、`visual_confidence_summary`、`alignment_status` 与 `eligible_for_precise_timing`；保留 `visual_observations` 作为兼容字段。
- `retrieval_index.json` 的每张 Card 新增轻量 `evidence_summary`，只含节点数、检测切段数、覆盖状态、对齐状态、精确时序资格与高/中/低置信度计数。
- 导出仍不携带媒体或完整文案。`observation_coverage != complete` 会阻断导出；`eligible_for_precise_timing = false` 会写入非阻断 warning，提醒下游不得把该视频用于精确时序学习。
